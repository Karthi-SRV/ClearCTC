## What this is

A two-phase compensation copilot for Indian software engineers switching jobs.

**Phase 1 — before HR talks:** enter current city, CTC, and expected increment → see equivalent salary and real living costs across 35+ Indian cities, adjusted for cost-of-living and family size. No AI call.

**Phase 2 — after offers arrive:** compare 2–3 real offers on actual take-home (new-regime tax, PF, gratuity, COL). Two flows:
- **Full comparison** — AI (Gemini) ranks offers, scores them, cites every claim.
- **Quick comparison** — deterministic snapshots + company details from DB. No AI call.

---

## Tech stack

- **Frontend:** React (Vite, TypeScript). Mobile-friendly. No `localStorage` / `sessionStorage` anywhere.
- **Backend:** NestJS (TypeScript). All AI calls happen server-side only.
- **Database:** MongoDB via Mongoose (`@nestjs/mongoose`). Redis (`ioredis`) for city expense caching.
- **AI:** Gemini free tier (company data + offer comparison), Ollama (city expenses by default). Selectable via `AI_PROVIDER` env.
- **Auth:** JWT (`@nestjs/jwt`), bcrypt. Global `JwtAuthGuard`; `@Public()` opts routes out.
- **Observability:** `prom-client` metrics, `nest-winston` logger, optional Loki transport, Grafana dashboard.

## Architecture — NestJS modules

### core/ (no controllers; export providers via DI tokens)

- **`CompensationModule`** — deterministic financial engine. Pure synchronous functions. No AI, no DB.
- **`DataModule`** — exposes `DATA_SOURCE` token. Active impl: `CachedDataSource` (reads `companies` collection + static COL index). Stub: `LiveDataSource` (throws on every method). Consumers inject the token, never the class.
- **`AiModule`** — three DI tokens:
  - `AI_CLIENT` — selected at runtime by `AI_PROVIDER` env (ollama / gemini / claude / google)
  - `CITY_EXPENSE_AI_CLIENT` — follows `AI_CLIENT` (same as `AI_PROVIDER`)
  - `COMPANY_AI_CLIENT` — always `GeminiAiClient`, regardless of `AI_PROVIDER`
- **`CityExpenseModule`** — city living costs. Lookup chain: Redis → MongoDB → AI fetch. Stores all 6 family-size breakdowns per city in one document.
- **`LoggerModule`** — `@Global()`. Winston with console + optional Loki transports.

### features/ (own ≥1 HTTP route + controller)

- **`AuthModule`** — signup / login (JWT). Stores user compensation profile.
- **`SalaryAskModule`** — Phase 1. Public endpoint. No AI.
- **`OfferComparisonModule`** — Phase 2 full. JWT. AI via `COMPANY_AI_CLIENT`.
- **`SalaryComparisonModule`** — Phase 2 lite. JWT. No AI.
- **`CityExpenseAdminModule`** — public city list + expense query + admin refresh.
- **`HealthModule`** — `GET /health`.

## Database — collections

| Collection | Purpose |
|------------|---------|
| `users` | Auth + compensation profile (currentCity, currentCtcLpa, basicPayLpa, variablePayLpa, isFixed, expectedHikePct, currentRole, preferredCities) |
| `companies` | Company data generated by Gemini: role benchmarks, ratings, reviews, aiProfile. Seeded names via `scripts/seed.ts`. |
| `city-expenses` | One document per city. All 6 family-size breakdowns stored together (individual, family 2–6). |
| `offers` | Raw offer inputs + computed snapshot. Never recomputed by AI. |

No `profiles` collection — user data lives in `users`.

## THE TRUST BOUNDARY (most important rule)

The AI reasons and explains. It **never** originates a fact with financial consequence.

| AI MAY | AI MAY NOT |
|--------|-----------|
| Rank/compare offers using numbers supplied to it | Compute take-home itself |
| Reason over salary benchmarks supplied to it | Invent a benchmark or salary figure |
| Summarise supplied review text and ratings | Invent reviews or ratings |
| Echo `CompensationService`-computed confidence, flag risk | Originate or modify confidence on any numeric figure |

All money math (CTC → in-hand, tax, PF, gratuity, COL) lives in `CompensationService` as pure functions with unit tests. The computed result is stored in `offers` and passed to the AI. The AI receives already-computed numbers and reasons over them.

## Hard rules

- `GEMINI_API_KEY`, `JWT_SECRET`, `MONGODB_URI` are server-side env only. Never in client code, never committed. `.env.example` documents all keys.
- AI response is validated (score bounds, breakdown sum, bestOffer name match) before being merged with deterministic data. 502 on validation failure. No raw model text ever reaches the client.
- No browser storage APIs (`localStorage`, `sessionStorage`, `IndexedDB`). All state in React memory.
- Consumers inject `DATA_SOURCE`, `AI_CLIENT`, `CITY_EXPENSE_AI_CLIENT`, `COMPANY_AI_CLIENT` tokens — never concrete classes.
- Every AI claim must cite which data field drove it (snapshot figure, benchmark, or review dimension).
- `CompensationService` owns confidence on all financial figures. AI echoes it; never re-derives it.
- Gemini free tier: 15 RPM. Company seed calls are sequential with 15 s gaps. Hard quota exhaustion (`[GEMINI_QUOTA_EXHAUSTED]` prefix) aborts the seed loop immediately — do not retry.
- `scripts/seed.ts` is idempotent — re-running never duplicates documents.

## Scope

**IN:** auth; Phase 1 city comparison; Phase 2 full (AI) + lite (deterministic); CompensationService; Redis city expense cache; family-size expense personalisation (individual / family 2–6); company data generated via Gemini at startup; failure-mode UX (stale-data confidence drop, fallback expenses).

**OUT:** live scraping (stub only), ESOP/RSU valuation, foreign offers, old tax regime, negotiation coaching.

Full rationale in `/docs/scope.md`.

## Coding conventions

- Small, pure, synchronous functions for all financial logic.
- NestJS module boundaries respected; communicate across modules via DI providers.
- Comments explain why, not what. No multi-line comment blocks.
- No error handling for scenarios that can't happen. Trust internal code and framework guarantees.
- TypeScript strict mode. No `any` except where Mongoose requires it.

| `/structured-ai-output` | Writing a new AI prompt or adding a validation rule |
